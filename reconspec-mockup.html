Endpoints

This API manages pet inventory, user accounts, and store orders. Authentication relies on API key and OAuth2 flows. The analysis identified several areas of concern, primarily around the user registration and authentication endpoints, which accept unvalidated input and lack rate limiting indicators in the spec. The pet management endpoints expose file upload functionality that warrants testing for injection and storage-based attacks.

Object-level authorization testing is recommended across all endpoints using path parameters, particularly {petId}, {username}, and {orderId}.

Injection via Name and Tag Fields
API8 Security Misconfiguration Params: name tag
Overview

The name and tag fields accept string input that may not be sanitized before storage or rendering. Test for SQL injection, NoSQL injection, and stored XSS depending on the backend technology.

Testing Steps
  1. Send a POST request with SQL injection payloads in the name field. Observe the response for error messages, time delays, or unexpected behavior that indicates query manipulation.
  2. Repeat with NoSQL payloads if the API appears to use a document database. Watch for operator injection via JSON structure manipulation.
  3. Submit XSS payloads and then retrieve the created resource via GET /pets. Check whether the payload is stored and rendered unescaped.
  4. Test with oversized strings (e.g., 10,000+ characters) to check for buffer handling issues or missing length validation.
